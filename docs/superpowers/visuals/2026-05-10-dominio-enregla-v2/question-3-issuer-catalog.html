Pregunta 3 · Emisor de permisos

¿Cómo modelamos al organismo que emite cada permiso?

Hoy permits.issuer es texto libre. Un mismo emisor aparece como "Bomberos Quito", "CBN Quito", "Cuerpo de Bomberos". No se agrupa, no se puede linkear a su portal, no se ve el logo. Lo ascendemos a entidad: nueva tabla permit_issuers.

B · Dos opciones para el modelo del catálogo

Recomendada

1 · Catálogo cerrado, editable solo por staff

Tabla permit_issuers con RLS staff-only para INSERT/UPDATE. El cliente solo puede elegir de la lista. Para agregar uno nuevo, abre ticket o lo hace EnRegla.

Pros

  • Data limpia y agrupable desde día 1.
  • Logos, URLs, ámbito consistente.
  • Matriz del Marco Legal siempre tiene emisor correcto.
  • Métricas "tienes N permisos con ARCSA" son confiables.

Contras

  • Si un cliente tiene un permiso con emisor raro (p.ej. un municipio chico), tiene que escalarnos.
  • Un poco de fricción inicial hasta que llenemos el catálogo en otras ciudades.

2 · Catálogo + escape hatch "otro"

Mismo catálogo base, pero el cliente puede seleccionar "Otro…" y escribir un emisor ad-hoc que queda como texto libre en el permit (no en el catálogo).

Pros

  • Nunca bloqueás al cliente — siempre puede continuar.
  • Ideal para corner cases (municipios chicos, organismos regionales).

Contras

  • Re-introduce el problema original: "CBomberos" vs "Cuerpo Bomberos" vs "Bomberos".
  • Métricas y matriz se vuelven inconsistentes para esos casos.
  • Probablemente nadie lo use — termina siendo complejidad inútil.

C · Schema propuesto (si elegís opción 1)

CREATE TABLE public.permit_issuers ( id uuid PRIMARY KEY DEFAULT gen_random_uuid(), slug text UNIQUE NOT NULL, -- 'sri', 'gad_quito', 'bomberos_quito' name text NOT NULL, -- 'Servicio de Rentas Internas' short_name text NOT NULL, -- 'SRI' scope text NOT NULL CHECK (scope IN ('nacional', 'municipal')), city text, -- 'Quito' si municipal, NULL si nacional portal_url text, -- link al portal de trámite logo_url text, -- opcional created_at timestamptz DEFAULT now() ); -- Luego cambiamos permits.issuer (text libre) → permits.issuer_id (FK) ALTER TABLE permits ADD COLUMN issuer_id uuid REFERENCES permit_issuers(id); -- Dejamos permits.issuer como string legacy durante la migración, después drop.

D · Beneficio concreto en UI

  • En el detalle del permiso: logo + botón "Ir al portal de trámite" → abre portal_url.
  • En la matriz del Marco Legal: "Bomberos — $50–$200 · emisor: Cuerpo de Bomberos de Quito" (clickeable).
  • Métricas futuras: "Tenés 3 trámites pendientes con ARCSA — bloque"; "Dedicaste 5 visitas al GAD este mes".
  • Si creamos soporte multi-ciudad después, cambiar de Quito a Guayaquil solo re-filtra el dropdown — el modelo soporta.

E · Mi recomendación

Opción 1 (catálogo cerrado, staff-only). Para MVP con 1 cliente en Quito, los 5 emisores cubren todo. El "escape hatch" de la opción 2 es complejidad preventiva sin caso real hoy. Si aparece el corner case, 2 minutos de SQL agregan el emisor nuevo.
