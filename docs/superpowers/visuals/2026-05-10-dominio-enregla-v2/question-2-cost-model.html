Pregunta 2 · Modelo de datos

¿Cómo guardamos el costo de un permiso?

Tres enfoques posibles. Elegimos uno (o combinado). Esto determina si el dashboard puede mostrar “te falta pagar $X” de forma confiable o si queda como adorno cualitativo.

A · Texto libre

Una columna de texto. Lo que hay hoy. Ej: "$50 aprox, varía por m²"
ALTER TABLE permit_requirements ADD COLUMN estimated_cost text; -- ejemplo: -- "$50-$200 según patrimonio"
Factura pendiente (Dashboard)
Bomberos"$50-$200 según patrimonio"
LUAE"Depende del patrimonio"
Patente"Variable"
Totalno calculable

Pros

  • Fácil: 1 columna nueva.
  • Cero decisiones técnicas de dominio.

Contras

  • Dashboard no puede sumar → la "factura pendiente" queda adorno.
  • Cada permit/giro tiene texto distinto → inconsistente.
  • Imposible hacer filtros tipo "permisos baratos" o "< $100".
Recomendada

B · Rango numérico + notas

Tres columnas: cost_min, cost_max, cost_notes. Si ambos son iguales, es costo fijo. Si son distintos, es rango. Las notas explican la variable.
ALTER TABLE permit_requirements ADD COLUMN cost_min numeric(10,2), ADD COLUMN cost_max numeric(10,2), ADD COLUMN cost_currency text DEFAULT 'USD', ADD COLUMN cost_notes text, ADD COLUMN cost_updated_at date; -- ejemplos: -- (0, 0, 'USD', NULL, '2026-01-15') <- RUC (gratis) -- (50, 200, 'USD', 'Varía por patrimonio') <- Bomberos -- (120, 120, 'USD', NULL) <- LUAE fijo Quito -- (NULL, NULL, 'USD', 'Depende del giro') <- desconocido
Factura pendiente (Dashboard)
Bomberos$50 – $200
LUAE$120
Patente~ $80
ARCSAsin estimar
Total estimado$250 – $400
+ 1 permiso sin costo estimado

Pros

  • Se puede sumar → dashboard muestra rango honesto.
  • Los permisos gratis quedan explícitos (0,0) vs los desconocidos (NULL).
  • cost_updated_at = puede mostrar "actualizado hace 3 meses" → confianza.
  • Permite filtros futuros sin migrar.

Contras

  • 3 columnas en vez de 1.
  • Hay que maintain los precios base (operación de contenido).

C · Fórmula dinámica

Costo se calcula a partir de datos del negocio (patrimonio, m², empleados). Columna cost_formula que se evalúa.
ALTER TABLE permit_requirements ADD COLUMN cost_formula jsonb; -- ejemplo Patente Municipal Quito: -- { "base": 0, "per_1000_patrimonio": 1.5, -- "min": 25, "max": 25000 } -- requiere nuevos campos en sede/company: -- patrimonio, m², empleados, facturacion
Factura pendiente (Dashboard)
Bomberos$85
LUAE$120
Patente$75
Total calculado$280
Según tu patrimonio declarado: $50,000

Pros

  • Número exacto (cuando la fórmula está bien).
  • Demo impresionante.

Contras

  • Cada municipio tiene fórmulas distintas → mantener 10 GADs es un dominio propio.
  • Requiere capturar datos adicionales del negocio (patrimonio real, m², etc) — fricción en onboarding.
  • Precisión falsa: si la fórmula está mal calibrada, el cliente cree $75 y paga $200 → pérdida de confianza mayor que rango.
  • YAGNI para MVP: demasiado para 1er cliente.

Mi recomendación: B

Rango numérico + notas es el balance correcto: suficientemente estructurado para sumar, suficientemente honesto para no prometer precisión que no tenés. El "total estimado $250 – $400" es más creíble que un número exacto mal calibrado (Opción C) o que un adorno inútil (Opción A). Si después del primer cliente ves que necesitás precisión por giro, se construye la Opción C encima — pero no antes.

Si elegís B, decisiones sub-menores:
  • Moneda: ¿Solo USD (Ecuador) o multi-moneda desde día 1? Mi voto: solo USD para MVP.
  • Precio por sede vs global: ¿Patente Municipal cuesta lo mismo en Quito que en Guayaquil? No. ¿Guardamos precio por (permit_type, city) o solo por permit_type con nota "Quito"? Mi voto: por ciudad ya desde MVP usando locations.city. No agrega complejidad y habilita Guayaquil sin re-migración.
  • Precios en la matriz del Marco Legal: la visualización pública del Marco Legal mostraría rango junto a cada permiso: "Bomberos — $50–$200". ¿OK?
  • Quién actualiza precios: staff-only via /internal/legal (futuro) o vos via SQL directo en prod. En ambos casos cost_updated_at deja huella.
