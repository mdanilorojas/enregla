El objetivo

Convertir EnRegla de un tracker con data mock a un producto que responde cuatro preguntas claras para cualquier dueño de PyME en Quito: ¿qué permisos debo tener · cuánto me cuesta ponerme al día · quién es responsable · qué pasa si no lo hago?

El diseño del dashboard no se toca. El trabajo es plomería de datos + schema. El diseño ya es bueno — le falta data real.

Modelo de datos

Nueva tabla permit_issuers

CREATE TABLE public.permit_issuers (
  id                     uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  slug                   text UNIQUE NOT NULL,
  name                   text NOT NULL,
  short_name             text NOT NULL,
  scope                  text NOT NULL CHECK (scope IN ('nacional', 'municipal')),
  city                   text,                        -- NULL si nacional
  portal_url             text,
  procedures_portal_url  text,
  contact_url            text,                        -- formulario web, no email
  phone                  text,
  address                text,
  notes                  text,                        -- ej: "Sede principal en Guayaquil" para ARCSA
  logo_url               text,
  created_at             timestamptz DEFAULT now(),
  updated_at             timestamptz DEFAULT now()
);

Extensión a permit_requirements

ALTER TABLE public.permit_requirements
  ADD COLUMN issuer_id        uuid REFERENCES permit_issuers(id),
  ADD COLUMN required_role    text NOT NULL DEFAULT 'anyone'
    CHECK (required_role IN ('anyone','representante_legal','contador','tecnico_responsable')),
  ADD COLUMN cost_min         numeric(10,2),
  ADD COLUMN cost_max         numeric(10,2),
  ADD COLUMN cost_currency    text DEFAULT 'USD',
  ADD COLUMN cost_notes       text,
  ADD COLUMN cost_updated_at  date,
  ADD COLUMN fine_min         numeric(10,2),
  ADD COLUMN fine_max         numeric(10,2),
  ADD COLUMN fine_source      text,             -- URL o cita legal
  ADD COLUMN applies_when     text;             -- descripción condicional (T)

Extensión a permits + profiles

ALTER TABLE public.permits
  ADD COLUMN issuer_id              uuid REFERENCES permit_issuers(id),
  ADD COLUMN assigned_to_profile_id uuid REFERENCES profiles(id) ON DELETE SET NULL;

ALTER TABLE public.profiles
  ADD COLUMN business_role text NOT NULL DEFAULT 'empleado'
    CHECK (business_role IN ('empleado','representante_legal','contador','tecnico_responsable'));

Nueva tabla permit_events (trazabilidad)

CREATE TABLE public.permit_events (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  permit_id   uuid NOT NULL REFERENCES permits(id) ON DELETE CASCADE,
  actor_id    uuid REFERENCES profiles(id) ON DELETE SET NULL,
  event_type  text NOT NULL CHECK (event_type IN (
    'created', 'status_changed', 'document_uploaded', 'document_deleted',
    'assigned', 'unassigned', 'renewed', 'dates_updated'
  )),
  from_value  text,
  to_value    text,
  metadata    jsonb,
  created_at  timestamptz DEFAULT now()
);

Bugs del dashboard que quedan arreglados

BugCausa raízFix
"Sin trámites pendientes" cuando sí los hay Código filtra solo por_vencer y vencido; ignora no_registrado y en_tramite. Filter: ['no_registrado','vencido','por_vencer','en_tramite'].
Factura muestra números mock Constante AVG_PERMIT_COST = 45 hardcoded. Suma real: Σ cost_min – Σ cost_max de pendientes, con JOIN a permit_requirements.
"Multa estimada" ficticia (multiplicador 3.2x/4.5x/6x) No hay dato real de multas. Nuevo seed con fine_min/fine_max + fine_source (link legal) por permiso.
Nombre de empresa = primera sede No se fetchea la company, solo las locations. Hook nuevo useCompany() o agregarlo al store.

Nota: las migraciones son atómicas y rollback-friendly. permits.issuer string queda deprecado pero vivo durante la migración — se elimina en un release siguiente cuando todos los datos estén en issuer_id.
