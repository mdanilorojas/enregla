1 · Problema y causa raíz

Síntoma actual

  • La app carga lento en la primera visita
  • A veces se queda en el skeleton loader para siempre
  • A veces "se muere" a mitad de uso — queries dejan de responder, no se muestra error
  • Es intermitente e impredecible

Causa raíz

~90% cliente

Los 8 hooks usan useState + useEffect + supabase.from() a mano. Sin caché, sin retry, sin timeout duro, sin dedupe.

Un solo glitch de red = skeleton permanente. Cada navegación entre rutas re-pega la DB sin razón.

~10% schema

Faltan índices en documents.permit_id y profiles.company_id. Helpers user_company_id() y user_role() marcados VOLATILE en lugar de STABLE.

Para 10 clientes casi no se nota, pero es barato de arreglar.

Auth layer acumuló 3 band-aids

  • Global flag authInitialized
  • Shared initializationPromise
  • Safety timeout de 5s que fuerza loading=false

Cada uno parcha un bug distinto de race conditions. Cuando metamos TanStack Query abajo, todos estos dejan de ser necesarios.

2 · Objetivos

Lo que debe lograr

  • Nunca skeleton permanente — si falla, UI lo muestra
  • Nunca query colgada > 10s sin feedback
  • Navegar entre rutas: instantáneo (cache)
  • Errores reales visibles en Sentry
  • PRs validados con CI antes de merge

Fuera de alcance

  • Rediseño de schema
  • Migración a otro backend
  • Edge Functions / SSR
  • Realtime subscriptions
  • Rewrite completo de RLS
  • Suite de tests E2E

4 · Arquitectura

Antes

Component → useLocations()
  → useState + useEffect
  → supabase.from()

- Sin cache
- Sin retry
- Sin timeout
- Re-fetch por mount

Después

Component → useLocations()
  → useQuery(['locations', id], fn)
  → QueryClient
  → supabase.from()

+ Cache compartido
+ Retry con backoff
+ Timeout 10s
+ Dedupe automático

QueryClient defaults

OptionValueEfecto
staleTime5 minDatos considerados frescos 5 min; no hay network request
gcTime30 minSe mantienen en memoria 30 min aunque no se usen
retry2Hasta 2 reintentos con backoff exponencial (1s, 2s)
networkMode'online'Sin red = no intenta query
refetchOnWindowFocustrueCambiar de pestaña y volver = revalida
refetchOnReconnecttrueVolver online = revalida
Custom wrapper10s timeoutAbortController + Promise.race; garantiza nunca colgarse

5 · Migración de hooks (big-bang)

Los 8 hooks migran en un solo commit con subagents paralelos. Uno por agente.

HookNueva keyNotas

useLocations['locations', companyId]Se desacopla de usePermits (era N+1 implícito)
usePermits['permits', { companyId, locationId }]

useDocuments['documents', permitId]

Mutaciones (updatePermit, renewPermit, uploadPermitDocument) usan useMutation y invalidan queries relacionadas en onSuccess → UI se actualiza sola.

6 · Manejo de errores

Componentes nuevos

  • src/components/ui/DataErrorBanner.tsx — se muestra en vez de skeleton cuando isError es true. Incluye botón Reintentar
  • src/components/ErrorBoundary.tsx — wrap global de <App>, captura errores de render, reporta a Sentry

Clasificación de errores

TipoUsuario veCliente hace
OfflineBanner naranja: "Sin conexión"Espera a online event para reintentar
Timeout 10sBanner rojo + ReintentarLog a Sentry
Auth 401/403Redirect a /loginLimpia Zustand + cache
Server 500Banner rojo + ReintentarRetry 2x auto + Sentry
RLS 403"No tienes permisos"Sentry, no reintenta

Sentry

  • Plan free (5K errors/mes, suficiente para 10 clientes)
  • Guarded por VITE_SENTRY_DSN — si no está, no-op
  • Captura: errores de render, promesas rechazadas, errores de TanStack Query
  • Excluye: session replay (feature de pago)
  • Contexto en eventos: user.id, company_id, pathname, última query key

7 · Cambios de DB

Migration 015 — índices faltantes

CREATE INDEX IF NOT EXISTS idx_documents_permit
  ON documents(permit_id);

CREATE INDEX IF NOT EXISTS idx_profiles_company
  ON profiles(company_id) WHERE company_id IS NOT NULL;

CREATE INDEX IF NOT EXISTS idx_permits_superseded_by
  ON permits(superseded_by) WHERE superseded_by IS NOT NULL;

Migration 016 — helpers STABLE

ALTER FUNCTION user_company_id() STABLE;
ALTER FUNCTION user_role() STABLE;

STABLE le dice a Postgres que el resultado es determinístico dentro de una query, habilitando caché. Solo mejora.

Types regeneración

npx supabase gen types typescript --local > src/types/database.ts

Para incluir notification_logs y notification_preferences que faltan.

Limpieza de artefactos

  • Borrar reset-demo-user.sql (bcrypt hardcoded)
  • Borrar scripts/create-demo-data.sql (referencia tabla dropped)

10 · Criterio de éxito

  • 30 min de uso real en el preview de Vercel sin un solo skeleton permanente
  • npm run build produce ≥ 5 chunks; bundle inicial < 400 KB
  • PR a main con CI verde (typecheck + lint + build + tests)
  • Sentry activo, capturando errores correctamente durante QA

Smoke test manual

  1. Abrir dashboard → datos visibles en < 2s
  2. Navegar entre rutas → instantáneo (cache)
  3. Matar Wi-Fi a mitad de query → ver banner con Reintentar
  4. Dejar pestaña idle 10 min → volver → revalida + datos frescos
  5. Abrir TanStack devtools → queries cacheadas visibles
